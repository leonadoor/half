# HALF 当前版本 PRD

> **对应范围**：当前仓库实现的 v0.x 早期开源版本。  
> **文档性质**：本文是面向产品范围、交互闭环和验收边界的 PRD；实现细节以代码、`docs/architecture.md` 和 `docs/task-lifecycle.md` 为准。

## 1. 产品定位

HALF（Human-AI Loop Framework）是一个人机协同多智能体任务管理控制台。它面向同时使用多个 AI coding agent（Claude Code、Codex、Copilot、GLM、Kimi 等）的研究者和团队，在合规前提下完成项目编排、Prompt 分发、Git 回写跟踪、任务状态推进和结果归档。

HALF 不直接运行 agent，不托管 agent 账号，不调用 agent 平台的非公开接口。系统只负责生成可人工复制的 Prompt，由项目负责人手工粘贴给外部 agent；agent 通过 Git 仓库回写计划和任务结果，HALF 后端通过轮询识别状态。

## 2. 当前阶段目标与非目标

### 2.1 目标

- 支持一个负责人用多个 AI coding agent 完成从项目创建、计划生成、任务派发到执行总结的闭环。
- 以 DAG 方式组织任务依赖，降低多 agent 协同中的人工调度和顺序管理成本。
- 通过 Git 协作仓库存储计划、任务产物、完成哨兵和可选用量文件，实现过程可追溯。
- 支持流程模板复用，减少常见协作流程的重复规划成本。
- 支持基础多用户、管理员配置、Agent 可用性管理和账号级资源隔离，适合自托管团队试用。
- 提供内置 Demo 项目和 Demo Agent，降低首次理解产品形态的成本。

### 2.2 非目标

- 不作为 Jira、Linear 或通用项目管理系统替代品。
- 不作为 agent runner，不自动调用、控制或模拟操作外部 agent。
- 不提供生产级多租户隔离承诺；当前部署模型以自托管、可信管理员为前提。
- 不支持网页对话类 agent 的自动上下文注入和自动执行。
- 不保证判断任务质量，只校验回写文件是否满足最小结构契约。

## 3. 用户与权限

### 3.1 角色

- **项目负责人**：创建项目、选择 Agent、生成计划、派发 Prompt、处理异常并查看总结。当前项目的所有人工操作均由负责人完成。
- **Agent 成员**：外部 AI coding agent，以 CLI 或用户交互工具形式执行任务，并按 Prompt 要求写回 Git。
- **管理员**：维护全局设置、Agent 类型与模型目录、公共 Agent、用户角色和账号状态。

### 3.2 账号体系

- 登录采用用户名和密码，密码规则为至少 8 位，包含大小写字母和数字。
- 系统初始化时通过 `HALF_ADMIN_PASSWORD` 创建默认 `admin` 账号。
- 自助注册由 `HALF_ALLOW_REGISTER` 控制，生产部署默认关闭。
- JWT token 有效期为 24 小时；冻结用户无法登录，已有 token 后续请求也会被拒绝。

### 3.3 权限边界

- 普通用户可管理自己创建的项目、私有 Agent、计划、任务和模板。
- 管理员可访问全局设置、Agent 设置和用户管理。
- 项目、计划、任务按 `created_by` 做 owner 级隔离，跨用户访问返回不存在或参数非法。
- Agent 采用“公共池 + 私有资源”模型：
  - 管理员创建的 Agent 是公共 Agent，活跃公共 Agent 对所有用户可见可用。
  - 普通用户创建的 Agent 是私有 Agent，仅创建者可见可用。
  - 公共 Agent 只能由创建它的管理员维护；其他用户只能使用活跃公共 Agent。
- 流程模板对所有登录用户可查看和使用，只有创建者和管理员可编辑或删除。

## 4. 核心对象

### 4.1 Project

Project 是 HALF 的协作单元，包含：

- 项目名称和目标说明。
- HALF 协作仓库地址 `git_repo_url`，必填，用于计划、任务结果、`result.json` 和可选 `usage.json` 的回写与轮询。
- 项目代码仓库地址 `project_repo_url`，可选，留空时等同于协作仓库；该地址进入任务 Prompt，但 HALF 不轮询该仓库。
- 协作目录 `collaboration_dir`，为仓库根相对路径；留空时系统生成默认目录。
- 参与 Agent 列表及每个 Agent 在本项目中的“同服务器”标记。
- 轮询区间、启动延迟、任务超时时间等项目级快照。
- 规划模式 `planning_mode` 和流程模板输入 `template_inputs`。

### 4.2 ProjectPlan

ProjectPlan 是计划记录。一个项目可有多个候选计划和一个最终计划。

- Prompt 路径计划写入 `<collaboration_dir>/plan-<plan_id>.json`，由 Git 轮询检测。
- 模板路径计划使用 `source_path = template:<template_id>`，应用模板后直接定稿，不走 Git 轮询。

### 4.3 Task

Task 由最终计划生成，形成 DAG。核心字段包括任务码、任务名称、描述、执行 Agent、前序依赖、预期输出说明、状态、超时时间和结果路径。

任务状态包括：

| 状态 | 含义 |
|---|---|
| `pending` | 已创建，尚未派发 |
| `running` | 已复制并派发，等待 Git 回写 |
| `completed` | 检测到合法 `result.json`，或被手动标记完成 |
| `needs_attention` | 超时、回写非法或 Git 同步错误，需要人工处理 |
| `abandoned` | 被负责人放弃；当前版本仍允许重新派发 |

### 4.4 Agent

Agent 是用户登记的外部 AI coding 工具。每个 Agent 包含名称、类型、模型列表、能力说明、默认同服务器标记、订阅到期时间、短期/长期重置时间、可用状态和排序信息。

Agent 可用状态采用四状态体验：

- 可用。
- 短期重置后可用。
- 长期重置后可用。
- 不可用，由订阅到期时间实时推导。

`chatgpt-pro` 类型 Agent 可在 Agent 页面通过 OpenAI OAuth 登录并刷新 Codex 额度；OAuth token、额度快照和刷新冷却只保存在后端进程内存中，不落库。

## 5. 核心使用流程

### 5.1 登录与初始化

用户访问系统后进入登录页。若部署方开启注册，用户可自助注册普通账号。首次启动默认创建 Demo 项目和 Demo Agents，用于展示项目看板、任务 DAG、Prompt 派发和 Agent 可用性。

### 5.2 创建项目

负责人创建项目时必须填写项目名称、HALF 协作仓库地址，并选择至少一个可用 Agent。项目代码仓库可单独填写；协作目录可手动指定或使用系统默认值。轮询参数在创建时从全局设置快照到项目，后续全局修改不追溯影响既有项目。

Git 仓库 URL 必须通过安全校验，拒绝本地地址、私网地址、危险协议、内嵌凭据、仓库内页面 URL 和可能导致命令注入的形式。

### 5.3 生成计划

Plan 页面提供两条路径。

#### 路径 A：使用流程模板

负责人选择模板，将模板中的 `agent-N` 抽象角色槽位映射到本项目的具体 Agent，并填写模板声明的必需输入。系统保存项目目标和模板输入后，直接把模板任务转换为最终计划并创建任务，然后进入任务执行页。

约束：

- 项目 Agent 数量必须满足模板角色数量。
- 槽位映射必须完整且不能重复。
- 必填模板输入为空时不能继续。
- 模板路径不生成规划 Prompt，也不启动计划轮询。

#### 路径 B：由 Prompt 生成

负责人填写任务介绍，选择规划模式，并勾选本轮参与规划的 Agent。每个 Agent 可手动指定模型；若未指定，系统按规划模式、Agent 类型说明和模型能力描述做软匹配。

规划模式包括：

- `balanced`：能力匹配与成本、速度的平衡。
- `quality`：优先效果，可增加并行候选和评审/合并任务。
- `cost_effective`：能力满足时优先节省用量或成本。
- `speed`：减少串行依赖，提高并发度并缩短关键路径。

点击“生成 Prompt”只生成 Prompt，不启动轮询。点击“拷贝 Prompt”成功写入剪贴板后，计划进入 `running`，后台开始轮询 `<collaboration_dir>/plan-<plan_id>.json`。系统检测到合法 JSON 后自动定稿并进入任务执行页。

同一未派发候选计划重复生成 Prompt 时复用同一 `plan_id` 和文件路径，避免已交给外部 agent 的路径失效。

### 5.4 任务执行

任务执行页以 DAG 展示任务依赖。负责人选择一个已解锁任务后，可复制 Prompt 并手工发送给对应 Agent。

派发规则：

- 前序任务必须全部 `completed` 或 `abandoned`，后继任务才可派发；后端在派发接口上强校验该约束。
- 前序未满足时，UI 上「复制 Prompt 并派发」和「放弃任务」按钮 disabled，避免误操作；放弃接口本身不做该校验。
- HALF 页面不检查前序产物是否真实存在；执行 Prompt 要求 agent 在本地 `git pull` 后自行检查。
- 任务名称、描述、预期输出被修改后，必须先自动保存并重新预取最新 Prompt，派发按钮才可用。
- “复制 Prompt 并派发 / 重新派发”必须保证剪贴板写入成功后才调用后端派发接口；复制失败时不得写入派发记录。

### 5.5 Git 回写与完成判定

Agent 执行任务后，应把所有协作产物写入：

```text
<collaboration_dir>/<task_code>/
```

所有产物写完后，最后生成并提交：

```text
<collaboration_dir>/<task_code>/result.json
```

`result.json` 是完成哨兵，必须是合法 JSON 对象，至少包含：

```json
{
  "task_code": "TASK-001",
  "summary": "本次任务的简要结果描述",
  "artifacts": ["outputs/proj-1/TASK-001/report.md"]
}
```

校验规则：

- `task_code` 必须等于当前任务码。
- `summary` 必须是非空字符串。
- `artifacts` 必须是仓库根相对路径字符串数组，不得使用绝对路径、反斜杠或 `..` 越界路径。
- 可选 `usage.json` 与 `result.json` 同目录；当前执行 Prompt 不再主动要求写入，但轮询检测到后会保存路径并在 UI 展示。

检测成功后任务进入 `completed`。检测失败或超时后任务进入 `needs_attention` 并展示 `last_error`。`needs_attention` 任务继续参与轮询，后续若合法结果出现，会自动恢复为 `completed` 并清除错误。

### 5.6 执行总结

总结页展示任务状态、执行 Agent、输出文件路径、完成时间和人工干预记录。当前版本以页面展示为主，不自动生成额外结构化报告文档。

## 7. 通知与设置

### 7.1 飞书通知

所有登录用户可配置个人飞书机器人 Webhook 和通知事件。通知按账户隔离，只发送用户自己创建项目/任务产生的事件。

支持事件：

- 任务完成 `completed`。
- 任务超时 `timeout`。
- 项目完成 `project_completed`。

Webhook URL 必须匹配 `https://open.feishu.cn/open-apis/bot/v2/hook/<token>`。推送失败只记录日志，不中断轮询。

### 7.2 全局项目参数

管理员可维护：

- 全局轮询最小/最大间隔。
- 轮询启动延迟。
- 默认任务超时时间。
- 规划 Prompt 中的同机分配引导文案。

项目创建时快照当前默认值；既有项目不受后续全局修改影响。

## 8. 非功能需求

### 8.1 合规与安全

- 全流程人工分发 Prompt，不自动调用外部 agent。
- 不抓取 agent Cookie，不读取非公开 token，不逆向外部协议。
- `HALF_SECRET_KEY` 和 `HALF_ADMIN_PASSWORD` 为必填安全项；严格模式下弱配置拒绝启动。
- Git URL、仓库路径和产物路径必须做安全校验，防止 SSRF、命令注入和路径越界。
- 审计日志记录用户角色变更、冻结/解冻和密码修改等敏感操作，不记录密码明文或哈希。

### 8.2 可用性

- 后端提供 `/health` 健康检查。
- Git 同步失败与“结果文件不存在”必须区分；远端不可达应写入可读错误，不得伪装为普通未完成。
- `git fetch` 成功但 `git pull --ff-only` 因分叉失败时，仅记录 warning，继续使用远端快照检测结果。
- 页面进入、窗口重新聚焦和手动刷新时拉取最新状态；后台轮询按项目配置执行。

### 8.3 性能体验

- 前端重页面使用路由级懒加载，避免 React Flow 等依赖拖慢首屏。
- 低频共享数据切回页面时优先显示最近成功内容，再后台刷新。
- 任务页和 Plan 页的 focus/visibility 刷新需要节流，避免重复请求。
- 支撑 DAG 和任务列表的后端接口应避免按任务数线性放大的重复 Git 操作。

## 9. 技术方案概要

| 层 | 选型 |
|---|---|
| 后端 | Python 3.12 + FastAPI + SQLAlchemy |
| 数据库 | SQLite |
| 认证 | JWT + bcrypt |
| 前端 | React 18 + TypeScript + Vite |
| 路由 | React Router |
| DAG | React Flow |
| Git 集成 | git CLI |
| 部署 | Docker Compose |

系统采用三层架构：浏览器负责页面交互和人工复制，FastAPI 后端负责鉴权、状态、Prompt、DAG 校验和 Git 轮询，SQLite 存系统状态，Git 协作仓库存项目协同产物。

## 10. 验收标准

当前版本满足以下条件视为核心闭环可用：

- 用户可登录系统，并在关闭注册时无法自助注册。
- 管理员可创建或维护公共 Agent、Agent 类型、模型目录和全局参数。
- 普通用户可使用公共 Agent 或自己的私有 Agent 创建项目。
- 项目可通过流程模板直接生成最终计划和任务。
- 项目可通过规划 Prompt 路径生成 `plan-<id>.json`，轮询检测后自动定稿。
- 任务可按 DAG 依赖逐步派发，复制失败不会写入派发记录。
- Agent 按固定目录写入合法 `result.json` 后，任务自动进入完成状态。
- 超时、非法 JSON、路径越界或 Git 同步错误能进入可见异常状态，并支持重新派发、手动完成、放弃。
- 执行总结能展示任务结果和人工干预记录。
- 飞书通知可按用户配置发送任务完成、任务超时和项目完成事件。

## 11. 版本边界与后续方向

当前 v0.x 已覆盖最小可用闭环、流程模板、多用户基础权限、公共/私有 Agent、飞书通知、Codex 额度辅助和安全硬化。

后续方向包括：

- 更完整的计划评审和多候选计划比较。
- 更细粒度的团队协作角色，而不仅是项目 owner。
- 更丰富的研究数据沉淀和实验分析。
- 更完整的通知与外部系统集成。
- 在合规前提下探索更多 agent 类型和执行环境适配。
